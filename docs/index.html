Interaction Benchmarks: React vs SolidJS

The following benchmarks measure real-world interaction performance — the metrics users actually feel. Use these to quantify the delta after migrating from React to SolidJS.

INP (Interaction to Next Paint)

Core Web Vital that measures the latency of all user interactions (clicks, taps, keypresses) throughout the page lifecycle. Reports the worst-case interaction (p98).

Target: < 200ms (good) · > 500ms (poor)
Measure: Chrome DevTools → Performance panel, or web-vitals library

TBT (Total Blocking Time)

Sum of all long tasks (>50ms) between FCP and TTI. Directly correlates with how "janky" the UI feels during load. SolidJS's lack of VDOM diffing dramatically reduces TBT.

Target: < 200ms (good) · > 600ms (poor)
Measure: Lighthouse, WebPageTest, or PerformanceObserver for long tasks

js-framework-benchmark

The gold-standard benchmark suite for comparing JS frameworks. Tests operations like create/update/swap/delete 1,000–10,000 rows in a keyed list.

SolidJS is consistently ~1.0x–1.06x vanilla JS speed
React is typically ~1.4x–1.8x vanilla JS speed
Source: krausest/js-framework-benchmark

Bundle Size

Smaller bundles mean faster parse/compile times which directly impact interaction readiness. SolidJS compiles away its reactivity — no runtime VDOM overhead.

solid-js: ~7 KB gzipped
react + react-dom: ~42 KB gzipped

Startup / Hydration Time

Time from script evaluation to interactive UI. SolidJS skips reconciliation — components run once, then only signals update the DOM surgically.

Measure: performance.mark() at app entry vs first interaction handler ready

Memory (Heap Snapshots)

SolidJS creates fewer JS objects (no virtual DOM tree, no fiber nodes). Lower memory pressure means fewer GC pauses and smoother interactions.

Measure: Chrome DevTools → Memory → Heap Snapshot, compare at idle after load

Expected Performance Delta

Metric React (typical) SolidJS (typical) Expected Improvement
Bundle size (gzip) ~42 KB (react+dom) ~7 KB ~83% smaller
INP (p75) 150–300ms 40–100ms 2–3x faster
TBT 200–500ms 20–80ms 5–10x lower
List ops (1k rows) ~1.5x vanilla ~1.03x vanilla Near-native speed
Memory at idle Baseline 30–50% lower Fewer GC pauses
Component mount Re-renders full tree Runs once, signals update DOM No re-render overhead

How to Measure Yourself

  1. web-vitals library — Drop-in measurement for INP, FCP, LCP, CLS, TTFB: 
    import { onINP, onTBT } from 'web-vitals';
onINP(console.log);  // logs every interaction latency
  2. Chrome DevTools Performance Panel — Record a session, check "Interactions" lane for INP-qualifying events and their processing duration.
  3. Lighthouse User Flows — Script a user flow (open window, drag, type in terminal) and measure TBT + INP across the entire interaction sequence: 
    import { startFlow } from 'lighthouse';
const flow = await startFlow(page, { name: 'Desktop Interaction' });
await flow.startTimespan({ stepName: 'Open Terminal' });
await page.click('[data-app="terminal"]');
await flow.endTimespan();
console.log(await flow.generateReport());
  4. js-framework-benchmark — Clone the repo and add Kasm's list component to get standardized row-operation timings: 
    git clone https://github.com/krausest/js-framework-benchmark
# Add kasm-solid/ and kasm-react/ framework entries
npm run build && npm run bench
  5. Custom marks — Instrument specific interactions: 
    performance.mark('window-drag-start');
// ... drag handler runs ...
performance.mark('window-drag-paint');
performance.measure('drag-latency', 'window-drag-start', 'window-drag-paint');
